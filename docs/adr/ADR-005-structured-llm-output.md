# ADR-005 — Structured Outputs для анализа LLM

Status: accepted (обновлён 2026-06-26: обработка нарушения лимитов длины — truncate вместо 502)

## Context

Финальный анализ должен иметь строгую структуру для отображения в iOS: `title` (≤3 слова), `overview` (≤40 слов), `advice` — массив секций `{heading, body}`. Нужен надёжный парсинг без хрупкого разбора свободного текста.

Отдельный вопрос — поведение при нарушении лимитов длины LLM-ответом. Изначально (bootstrap) было решено: 1 retry → `502 llm_upstream_error`. На практике это означает, что вся запись не финишируется (очки не начисляются, streak не обновляется) из-за косметического превышения — LLM выдал заголовок в 4 слова вместо 3. Это плохой UX: пользователь прошёл весь флоу, но финал падает с ошибкой вышестоящего провайдера, хотя контент по сути валиден.

## Decision

- Анализ (LLM#2) запрашивается через **OpenAI Structured Outputs** (`json_schema`, strict). Схема — `app/llm/analysis_schema.py`: `{title, overview, advice:[{heading, body}]}`.
- Follow-up (LLM#1) — обычный текстовый ответ (короткий вопрос).
- **Обработка лимитов длины (`title` ≤ 3 слова, `overview` ≤ 40 слов) — Вариант A (мягкий обрез):**
  1. Лимиты валидируются после генерации.
  2. При нарушении — **1 retry** (полная перегенерация), даём LLM шанс уложиться сам.
  3. Если после retry лимит всё ещё превышен — **мягкий обрез (truncate)** до лимита по словам (`title` → первые 3 слова, `overview` → первые 40 слов) и `finish` **завершается успешно** (`200`). Запись финишируется, очки начисляются, streak обновляется.
  4. `502 llm_upstream_error` остаётся **только** для реальных ошибок провайдера: ошибка API, пустой/невалидный JSON, недоступность модели. Косметическое превышение длины больше не приводит к `502`.
  5. Факт обреза логируется (`analysis_length_truncated`) для наблюдаемости качества промтов.
- Тексты промтов фиксируются **дословно** в `app/llm/prompts/v1.py` с `PROMPT_VERSION`. Дословные формулировки — в [modules/entries](../modules/entries/README.md).

## Consequences

- (+) Детерминированный парсинг, меньше ошибок интеграции с iOS.
- (+) Версионирование промтов (`prompt_version` сохраняется в БД).
- (+) `finish` не падает из-за косметического превышения — лучший UX, запись всегда финишируется при валидном по сути ответе.
- (−) Возможен обрезанный по середине заголовок/overview в редких случаях; компенсируется retry и наблюдаемостью (`analysis_length_truncated`).
- (−) Зависимость от поддержки Structured Outputs выбранной моделью (Q-LLM-1).

## Alternatives

- Вариант B: 1 retry → `502`. Отклонён: ломает завершение записи из-за косметики, теряет начисление очков/streak, плохой UX.
- Парсинг свободного текста по разделителям — отклонено: хрупко, ломается на разных языках.
