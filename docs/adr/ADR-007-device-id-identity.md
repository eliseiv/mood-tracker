# ADR-007 — Анонимная идентификация по device-id (опаковая строка)

Status: accepted
Дата создания: bootstrap; пересмотрено iteration 8 (UUID → произвольная непустая строка).

## Context

Приложение анонимное, без регистрации и PII. Нужен способ привязать данные к устройству и скоупить доступ. Изначально `X-Device-Id` требовался строго в формате **UUID v4** (middleware валидировал формат). На практике это усложняет интеграцию клиента и ручное тестирование: разработчику/тестеру нельзя послать читаемый id вроде `testuser` — middleware отвечает `400 device_id_invalid`. Требование пользователя: разрешить `X-Device-Id` как **произвольную непустую строку**, сохранив анонимность и модель скоупа.

## Decision

`X-Device-Id` — **опаковый строковый идентификатор**, задаваемый клиентом (необязательно UUID):

- Заголовок `X-Device-Id` обязателен на всех `/api/v1/*`, кроме `GET /health`. Проверяется **после** app-level `X-API-Key` (ADR-009).
- Значение — **непустая строка** в пределах правил валидации (ниже). UUID остаётся валидным частным случаем.
- **Хранится как есть** (опаково): сервер не маппит его в UUID и не нормализует регистр. `GET /me` возвращает `device_id` **ровно той строкой**, что прислал клиент (после trim). Прислал `testuser` → видит `testuser`.
- Middleware: app-key → валидация `X-Device-Id` → upsert `Device` (создаёт при первом обращении, обновляет `last_seen_at`) → проброс `device_id` в request scope.
- Все данные скоупятся по `device-id`. Доступ к чужому ресурсу → `404` (не `403`).
- Логина нет. Переустановка приложения = новый device-id (Q-ID-1); клиент хранит id в Keychain для устойчивости (рекомендация сохраняется и для строковых id).

### Правила валидации `X-Device-Id`

- Сервер **обрезает окружающие пробелы** (trim); далее валидирует и хранит результат trim.
- **Непустой** после trim. Пусто/только пробелы → `400 device_id_required`.
- **Длина** ≤ **64** символа (после trim). Превышение → `400 device_id_invalid`.
- **Charset**: ASCII буквы/цифры и `-`, `_`, `.` (regex `^[A-Za-z0-9._-]+$`). Пробелы внутри, управляющие и любые иные символы (включая не-ASCII) запрещены. Нарушение → `400 device_id_invalid`.
- **Case-sensitive**: `TestUser` и `testuser` — разные устройства (как и UUID разного регистра). Нормализация не выполняется.

Выбор `64` — с запасом покрывает UUID (36 символов) и читаемые id; ограничивает раздувание индекса/строки и закрывает инъекции/лог-форджинг. Charset `[A-Za-z0-9._-]` безопасен для использования в логах, заголовках и индексах.

## Consequences

- (+) Нулевой friction онбординга, отсутствие PII, простая модель доступа; проще интеграция и ручное тестирование (`testuser`).
- (+) Модель остаётся анонимной: id опаков, скоуп по нему, app-level `X-API-Key` по-прежнему гейтит доступ приложения.
- (−) Потеря device-id (без Keychain) = потеря доступа к данным. Документировано для iOS.
- (−) Нет кросс-девайс синхронизации — вне scope.
- (−) **Тип `Device.id` меняется UUID → `String(64)`** (и все FK-колонки). Миграция в проде — см. [03-data-model.md](../03-data-model.md). После появления не-UUID id обратный downgrade (`varchar→uuid`) **становится невозможен** — осознанное одностороннее последствие.
- (−) Клиент сам отвечает за уникальность/стабильность своего id (как и раньше с UUID); коллизия строк = общий доступ к данным. Для prod-клиента остаётся рекомендация генерировать стабильный уникальный id (UUID) и хранить в Keychain.

## Alternatives

- **Оставить строго UUID v4** — отклонено: не удовлетворяет требованию (читаемые/произвольные id для интеграции и тестов).
- **Принимать строку, но хранить хеш/маппинг в UUID** — отклонено: скрытый маппинг ломает echo «прислал → увидел то же», усложняет отладку без выгоды (id и так опаков).
- **Детерминированная UUIDv5-деривация из строки** (`uuid5(namespace, x_device_id)`) — отклонено: это частный случай скрытого маппинга. Хранимый/возвращаемый id перестаёт совпадать с присланным — `GET /me` вернул бы выведенный UUID, а не `testuser`, нарушая требование echo. Деривация не даёт выгоды (опаковость и скоуп уже обеспечены), но добавляет неинтуитивное преобразование и теряет читаемость id в логах/отладке.
- **Полноценная авторизация (OAuth/email)** — отклонено: противоречит анонимности и усложняет продукт.
- **Идентификация по IDFV/IDFA** — отклонено: нестабильно, ограничения приватности платформы.
