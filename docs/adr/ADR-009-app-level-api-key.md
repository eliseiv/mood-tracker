# ADR-009 — App-level аутентификация по статическому API-ключу

Status: accepted
Дата: 2026-06-26 (iteration 5); дополнено iteration 6 (семантика включения барьера + prod-guard `APP_ENV`)

## Context

До сих пор API было анонимным: единственный заголовок `X-Device-Id` (ADR-007) идентифицирует устройство/пользователя, но **не подтверждает, что запрос исходит от легитимного приложения**. Любой внешний клиент, знающий формат API, мог обращаться к `/api/v1/*` (создавать device-id и расходовать дорогие LLM/STT-вызовы в пределах per-device rate limit). Требуется app-level барьер: отсекать публичный доступ/абьюз ещё до анонимной идентификации устройства.

Полноценная пользовательская аутентификация (аккаунты, OAuth/JWT) продукту на старте не нужна — приложение анонимно по дизайну (ADR-007). Нужен простой общий секрет приложения.

## Decision

Вводится **app-level аутентификация по статическому общему API-ключу**:

- Заголовок **`X-API-Key`** — статический общий секрет приложения (один на все инсталляции).
- **Обязателен на всех `/api/v1/*`**. Единственное исключение — `GET /health` (без `X-API-Key` и без `X-Device-Id`, для liveness/readiness инфраструктуры).
- Секрет хранится в env-переменной **`API_KEY`** (pydantic-settings), наравне с `OPENAI_API_KEY`: никогда в коде/репозитории/логах; в prod — через secret manager; `.env.example` содержит `API_KEY=` без значения.
- **Порядок проверки в middleware: СНАЧАЛА `X-API-Key`, ПОТОМ `X-Device-Id`.** Запрос без/с неверным ключом отклоняется до валидации device-id и до любого upsert `Device`.
- Коды ошибок (единый формат `{error:{code,message,details?}}`):
  - отсутствует `X-API-Key` → `401 api_key_required`;
  - неверный `X-API-Key` → `401 api_key_invalid`.
  - `401` отдаётся без раскрытия деталей (не сообщать, ключ ли неверен по длине/префиксу и т.п.).
- Сравнение ключа — **constant-time** (например `secrets.compare_digest`) для защиты от timing-атак. Ключ не логируется.

`X-API-Key` **не заменяет** `X-Device-Id`: первый аутентифицирует **приложение** (общий секрет), второй идентифицирует **устройство/пользователя** анонимно (ADR-007). Оба обязательны вместе на `/api/v1/*`.

Один статический ключ — осознанное упрощение для MVP. Ротация/набор per-client ключей — возможное расширение (см. [Q-AUTH-1](../99-open-questions.md#q-auth-1)), не блокер.

### Семантика включения барьера и prod-guard (fail-safe)

Реализация enforced-логики: проверка `X-API-Key` включается **тогда и только тогда, когда env `API_KEY` непустой**. Пустой `API_KEY` → проверка пропускается (барьер выключен) — это нужно для локальной разработки/тестов и обратной совместимости. Чтобы это не противоречило «`X-API-Key` обязателен» и не превратилось в тихий fail-open в проде, вводится явная среда и guard:

- **В PROD `API_KEY` ОБЯЗАТЕЛЕН.** Деплой обязан задать непустой `API_KEY`; его отсутствие в prod = ошибка конфигурации, а не «открытый доступ».
- **Пустой `API_KEY` допустим ТОЛЬКО для local/dev/test** (барьер сознательно выключен; per-device rate limit и прочие меры сохраняются).
- **Prod-guard (fail-closed, требование к backend):** вводится дискриминатор среды env **`APP_ENV`** (`local` | `prod`, default `local`). При `APP_ENV=prod` **и** пустом `API_KEY` приложение **отказывается стартовать** (явная ошибка конфигурации на старте), а не запускается с выключенным барьером. При `APP_ENV=local` пустой `API_KEY` допустим (барьер выключен, лог-предупреждение). Когда `API_KEY` непустой — барьер enforced в любой среде.

**Почему guard, а не только runbook.** Тихий fail-open (prod без ключа = полностью открытый API) невидим в рантайме: всё «работает», но без app-level аутентификации. Полагаться только на чеклист деплоя для security-критичного барьера ненадёжно (человеческая ошибка не детектируется). Стартовый guard превращает misconfiguration в громкий отказ деплоя (fail-closed) ценой одного env-флага. Deployment-checklist сохраняется как defense-in-depth, но не как единственная мера. Стоимость guard минимальна, поэтому он принят как обязательное требование к backend (не TD/Q).

## Consequences

- (+) Публичный доступ к API закрыт app-level барьером поверх per-device rate limit — снижает абьюз дорогих LLM/STT-вызовов вне легитимного клиента.
- (+) Минимальная сложность: один заголовок, один env-секрет, constant-time сравнение; не вводит аккаунты/сессии/токен-инфраструктуру.
- (+) Ранний отказ (`401` до проверки device-id) экономит ресурсы и не создаёт `Device` для нелегитимных запросов.
- (+) Prod-guard (`APP_ENV=prod` + пустой `API_KEY` → отказ старта) делает misconfiguration громкой (fail-closed), исключая тихий fail-open в проде; локальная разработка не страдает (пустой ключ допустим при `APP_ENV=local`).
- (−) Статический общий ключ компрометируем (его можно извлечь из клиента); защищает от случайного/массового абьюза, но не от целевого реверса. Митигизация частичная: ключ как секрет, TLS, per-device rate limit; ротация — расширение.
- (−) Ротация ключа требует одновременного обновления клиента и сервера (без набора ключей — короткое окно несовместимости). Принято для MVP; решается per-client ключами позже.

## Alternatives

- **OAuth2 / JWT с аккаунтами** — отвергнуто для MVP: продукт анонимен (ADR-007), пользовательских аккаунтов нет; вводит непропорциональную сложность.
- **Per-client / ротируемый набор ключей** — отложено (Q-AUTH-1): ценно для ротации без простоя, но избыточно на старте.
- **mTLS / подпись запросов** — отвергнуто: сложно для iOS-клиента и инфраструктуры, непропорционально задаче «отсечь публичный доступ».
- **Без app-key (только device-id)** — отвергнуто: не защищает от внешнего абьюза.
- **Fail-safe только через deployment-runbook (без prod-guard)** — отвергнуто как единственная мера: чеклист не детектирует человеческую ошибку, тихий fail-open в проде недопустим для security-барьера. Runbook оставлен как defense-in-depth поверх guard.
